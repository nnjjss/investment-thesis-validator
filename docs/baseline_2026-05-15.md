## What the numbers mean

- **`final_answer_accuracy = 0.800`** — 4 out of 5 stances correct. The miss
  (`f01`, AAPL TTM FCF margin > 25%) is interesting because the model had
  income statements and could compute FCF directly. Manual investigation
  needed — likely a prompt issue or numerical-reasoning weakness.
- **`tool_call_accuracy = 0.700`** is depressed by *infrastructure*, not
  agent behavior:
  - `s04` needs news, `news_api_key` is unconfigured → tool 0.00
  - `m01` planner chose stock-only despite news being expected → 0.50
- **`retrieval_precision = 0.397`** is low because the seed's
  `expected_evidence_keys` use loose prefixes and FMP `/ratios` is
  paywalled on the free tier (so returned evidence is exclusively profile
  + quote + income, while ratios prefixes never match). Tighten the seed
  expectations or upgrade FMP for a fair number.
- **`hallucination_rate = 0.000`** — the synthesize node's evidence-id
  cross-check is doing its job. Model never cited an id that wasn't in
  the bundle.
- **Cost**: $0.0588 per validation. Synthesize is the bulk (>80%); cache
  reads were 0 across the run because each item has a fresh evidence
  bundle, so caching only kicks in within a single graph traversal.

## Known limits this run is exposed to

1. FMP free tier blocks `/ratios` (HTTP 402). `fetch_stock` now isolates
   per-endpoint failures so the run still produces evidence; `ratios` data
   is just absent until upgrade.
2. `NEWS_API_KEY` not configured. `fetch_news` returns `status=skipped`.
   Sentiment items report tool_acc=0 because of this — the metric is
   correctly honest about coverage gaps.
3. Sample of 5 — too small to draw confident conclusions about the
   distribution. Re-run at full 20 (or 100 after generator expansion)
   before invoking the M5 compile loop.
